# Dober 記事執筆スキル

アウトライン1章分 + リサーチ結果を受け取り、Doberブランドの本文を執筆する。
1セクション = 1回の実行。

## 前提

作業開始前に必ず参照:
- `assets/brand-guideline.md` — ボイス・トーン・禁止表現・フォーマット

## インプット

| 項目 | 必須 | 説明 |
|------|:----:|------|
| セクション情報 | ○ | アウトラインの対象章（タイトル・目的・キーポイント・トーン） |
| リサーチ結果 | △ | dober-article-searchの出力。「要リサーチ: なし」の章は不要 |
| 既執筆セクション | △ | 前章の末尾200字程度。文体・流れの一貫性を保つため |

## 記事の文体について（Threadsポストとの違い）

Threadsポストは断言・煽り・短文が主戦場。記事・LMは別のモードで書く。

| 項目 | Threadsポスト | 記事・LM |
|------|--------------|---------|
| トーン | 断言・叱責・危機感 | 解説・分析・教育的 |
| 文体 | 短文・箇条書き中心 | 段落・フレームワーク |
| 根拠 | 格言・体験談 | 著名人の実例・研究・統計 |
| 締め方 | 行動命令・喝 | 希望・次章への橋渡し |
| 解釈の余地 | ほぼなし | 意図的に残す |

## 実行手順

### Step 1: 執筆前の確認

以下を把握してから書き始める:

- **トーンレベル**: アウトライン指定（🔴最強 / 🟠強 / 🟡中 / 🟢低）
- **この章の目的**: 読者に何を理解させ、何を感じさせるか
- **使うエビデンス**: リサーチ結果・著名人事例・著者自身の体験から選ぶ
- **前章との接続**: 前章の流れを受けて自然につながるか

### Step 2: 本文執筆

**文章の基本構造:**
```
[つかみ — 問いかけ・逆説・リアルな事例で読者を引き込む]

[概念定義 — フレームワークや新しい視点を提示する]

[本論 — キーポイントを順番に展開]
  - 主張 → 著名人事例 or 研究 or 著者体験 → 読者への示唆
  - 遺伝的要素 vs 後天的要素など、構造化して整理する

[セクション締め — 希望・次章への橋渡し・読者への問いかけ or ワーク]
```

**エビデンスの使い方（優先順）:**

1. **著名人・有名人の実例** — 「エンドリックは〜」「フジモンは〜」のように固有名詞で具体性を出す
2. **著者自身の体験談** — 「私自身〜だった」から「今では〜できるようになった」の変化の物語
3. **研究・統計** — 数値で裏付けるが、断言よりも「〜という研究がある」「〜と言われている」と解釈の余地を残す
4. **論理的な因果関係** — 「〜だからこそ〜なのだ」と読者が自分で納得できる構造

**フレームワーク思考の活用:**
複雑な概念を整理するとき、以下の対比構造を使う:
- 遺伝的要素 vs 後天的要素
- 固定値 vs 変動値
- 世間の認識 vs 実態
- 短期 vs 長期

**トーンレベル別の書き方:**

| レベル | 語尾・表現の傾向 | 使う感情 |
|--------|----------------|---------|
| 🔴最強 | 〜だ。〜している。現実を鋭く突きつける | 羞恥・危機感 |
| 🟠強 | 〜になる。〜を失う。失敗パターンを示す | 恐怖・警告 |
| 🟡中 | 〜できる。〜が鍵だ。フレームワークを提示 | 覚悟・理解 |
| 🟢低 | 〜と言える。〜ではないだろうか。解釈を促す | 納得・希望 |

記事では🟡中〜🟢低が主軸。🔴🟠は導入や問題提起の場面に限定する。

**必ず守るルール:**
- エモジ禁止・ハッシュタグ禁止
- 「みなさん」「参考になれば幸いです」などの柔らかすぎる表現禁止
- 抽象論で終わらず、著名人事例・体験談・数字で具体化する
- 読者への問いかけ・自己採点ワークなど「参加させる」仕掛け、CTAを章末に入れる
- 記号は `・` `→` `①②③` `**太字**` のみ

### Step 3: 出力フォーマット

```markdown
## [セクションタイトル]

### [サブセクション名]（必要な場合）

[本文]

---
<!-- 執筆メタ情報（次セクション執筆時に削除） -->
- 文字数: 約XXX字
- 使用エビデンス: [使用した事例・統計・引用]
- 次章への接続ポイント: [次のセクションが受け取るべき文脈]
```

### Step 4: 自己チェック

出力前に以下を確認する:

**必須チェック:**
- [ ] エモジ・ハッシュタグが混入していないか
- [ ] 抽象論だけで終わっている段落がないか（事例・数字・体験で具体化されているか）
- [ ] トーンレベルが指定通りか（記事は🟡〜🟢が主軸）

**品質チェック:**
- [ ] つかみで読者が「自分のことだ」と感じられるか
- [ ] 著名人事例 or 著者体験が少なくとも1つ入っているか
- [ ] フレームワーク・対比構造で概念が整理されているか
- [ ] 章末に「読者が自分ごと化できる問いかけ or ワーク」があるか
- [ ] 次章への橋渡しが自然か

## エラーハンドリング

| 状況 | 対処 |
|------|------|
| リサーチ結果に「情報不足」とある | 著名人の実例 or 著者体験談で代替。それも不明なら「要確認」と明記してユーザーに確認する |
| 前章の文体と乖離している | 前章末尾200字を再確認し、語尾・テンポを揃える |
| 指定文字数を大幅に超過 | サブセクションのどれかを圧縮。フレームワーク説明を箇条書きに切り替える |
| 指定文字数を大幅に下回る | 著名人事例を追加するか、著者体験の変化ストーリーを補完する |
